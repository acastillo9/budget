# Code Review: Notification System & Multi-Channel Alerts

## Critical

- **[budget-api/src/notifications/channels/email.channel.ts:33]** *(Error Handling / Runtime Crash)* -- `MailService.sendMail()` calls `I18nContext.current()` internally (in `mail.service.ts:16`), which returns `null` when there is no active HTTP request context. Email notifications sent from cron jobs (`BillScannerJob`, `BudgetCheckerJob`, `LowBalanceJob`, `MonthlySummaryJob`) and event handlers (`TransactionNotificationHandler`) will throw `Cannot read properties of null (reading 'lang')` at runtime because `I18nContext.current()` is `null` outside of request scope. **Fix:** Either pass a language/locale in the `NotificationEvent` interface and have `EmailChannel` set `context.i18nLang` explicitly before calling `MailService.sendMail()`, or modify `MailService.sendMail()` to gracefully handle a null `I18nContext.current()` (e.g., `I18nContext.current()?.lang || 'en'`). The latter is safer as it also protects other future non-request callers.

- **[budget-api/src/notifications/channels/in-app.channel.ts:14-31]** *(Code Quality / Redundant Data)* -- `InAppChannel.send()` passes `user` and `workspace` both inside the `CreateNotificationDto` object AND as separate `userId`/`workspaceId` arguments to `NotificationsService.create()`. The `create()` method then overwrites `dto.user` and `dto.workspace` with the separate arguments (lines 32-36 of `notifications.service.ts`). This means the `user` and `workspace` fields in the DTO are always overridden and serve no purpose. While not a bug (the correct values end up in the database), it is confusing and could lead to bugs if someone trusts the DTO values. **Fix:** Remove `user` and `workspace` from the DTO argument in `InAppChannel.send()`, or remove the `userId`/`workspaceId` parameters from `NotificationsService.create()` since the DTO already carries them.

- **[budget-api/src/notifications/jobs/bill-scanner.job.ts:29]** *(External Config)* -- `@Cron(process.env.CRON_BILL_SCANNER || '0 8 * * *')` uses `process.env` directly instead of `ConfigService`. The `@Cron` decorator evaluates at class definition time (before NestJS DI container is fully initialized), so `ConfigService` cannot be injected here. However, this applies to all 5 cron jobs and is a known NestJS limitation. The current `process.env` fallback approach is actually the standard pattern for `@nestjs/schedule`. Downgrading to **Warning** -- see Warning section.

## Warning

- **[budget-api/src/notifications/jobs/bill-scanner.job.ts:29,47]** *(Performance / N+1 Queries)* -- The bill scanner job queries ALL bills (`this.billModel.find({})`), then for each bill queries all workspace members, then for each member queries their preferences individually. For a workspace with 100 bills and 5 members, this produces 100 + 100 + 500 = 700 queries. Similar N+1 patterns exist in `BudgetCheckerJob` (line 45) and `LowBalanceJob` (line 39). **Fix:** Batch-load members and preferences per workspace before iterating. Group bills by workspace, load members once per workspace, and pre-load all preferences for those members in a single query.

- **[budget-api/src/notifications/handlers/transaction-notification.handler.ts:77-81]** *(Performance / Duplicate DB Queries)* -- `checkBudgetThresholds`, `checkLargeTransaction`, and `checkLowBalance` all independently query `NotificationPreference` for the same user+workspace combination. Since they are called via `Promise.allSettled`, this results in 3 parallel queries for the same document. **Fix:** Query preferences once in `handleTransactionCreated` and pass the result to all three check methods.

- **[budget-api/src/notifications/handlers/transaction-notification.handler.ts:83-98]** *(Code Quality / DRY)* -- `getPeriodStart` and `addPeriod` helper methods are duplicated identically in both `TransactionNotificationHandler` and `BudgetCheckerJob`. **Fix:** Extract these into a shared utility (e.g., `src/notifications/utils/period.util.ts`) or a shared service.

- **[budget-api/src/notifications/dto/update-notification-preference.dto.ts:22-23]** *(Security / Validation)* -- The `channels` field is validated only as `@IsObject()`, but the keys and nested values are not validated. A client could send arbitrary keys (e.g., `channels: { "ARBITRARY_KEY": { inApp: true, email: true } }`) and they would be persisted to MongoDB via the `$set` deep merge. While this does not cause a security vulnerability per se (the keys are only used for notification routing), it pollutes the database. **Fix:** Add a custom validator that validates channel keys against the `NotificationType` enum, and validates nested values have `inApp: boolean` and `email: boolean`.

- **[budget-api/src/notifications/jobs/notification-cleanup.job.ts:30-31]** *(External Config)* -- `this.configService.get('NOTIFICATION_RETENTION_DAYS', '90')` uses `get()` with a default string value instead of `getOrThrow()`. Per CLAUDE.md convention, mandatory env vars should use `getOrThrow()` in the constructor. However, since this has a reasonable default (90 days) and is not truly mandatory, using `get()` with a default is acceptable. The real issue is that `parseInt` on a potentially `undefined` value could yield `NaN` if the env var is set to a non-numeric value. **Fix:** Add a `parseInt` fallback: `parseInt(..., 10) || 90`.

- **[budget-api/src/notifications/entities/notification-lock.entity.ts:18]** *(Database)* -- The `NotificationLock` schema has an `expiresAt` field but no MongoDB TTL index on it. The lock service relies on checking `expiresAt < now` in the `acquireLock` query, but expired lock documents will accumulate in the collection forever. **Fix:** Add a TTL index: `.index({ expiresAt: 1 }, { expireAfterSeconds: 0 })` to the schema to let MongoDB automatically clean up expired locks.

- **[budget-api/src/notifications/notifications.service.ts:56]** *(TypeScript)* -- `const filter: any = { ... }` uses `any` type. Similar `any` casts appear in `updatePreferences` (line 234), `transaction-notification.handler.ts` (line 87, 120), and `budget-checker.job.ts` (line 60, 97). **Fix:** Define proper filter type interfaces (e.g., `FilterQuery<Notification>` from Mongoose) instead of using `any`.

- **[budget-ui/src/routes/(app)/+layout.svelte:62-67]** *(Performance)* -- The unread count polling `$effect` calls `fetchUnreadCount()` immediately on mount plus every 60 seconds. If the user is on a slow connection or the API is down, the 60-second interval continues regardless. Also, when the panel is open, the data shown could be stale while the polling only updates the count. **Fix:** Consider pausing polling when the document is not visible (using `document.hidden`) and refreshing the full notification list when the panel is opened.

- **[budget-ui/src/lib/components/notification-item.svelte:19]** *(Code Quality)* -- `dayjs.extend(relativeTime)` is called at module scope every time the component is imported. While dayjs handles idempotent extensions gracefully, it is better practice to extend it once at app initialization. **Fix:** Move `dayjs.extend(relativeTime)` to the app-level initialization (e.g., in the root layout or a setup module).

## Suggestion

- **[budget-api/src/notifications/notifications.service.ts:4]** *(Code Quality)* -- `plainToClass` from `class-transformer` is deprecated in favor of `plainToInstance`. The rest of the codebase also uses `plainToClass`, so this is a pre-existing pattern, but new code should prefer `plainToInstance`. **Fix:** Replace `import { plainToClass } from 'class-transformer'` with `import { plainToInstance } from 'class-transformer'` and update all call sites.

- **[budget-api/src/notifications/handlers/transaction-notification.handler.ts:14-21]** *(Conventions)* -- The `TransactionCreatedEvent` interface is defined inline in the handler file with `transaction: any`. This should be a properly typed interface in a shared location (e.g., `src/notifications/services/` alongside `NotificationEvent`), and `transaction` should use `TransactionDto` or a specific type instead of `any`. Same for `WorkspaceInvitationCreatedEvent` (workspace-notification.handler.ts:7-12) where `invitation: any`. **Fix:** Create shared event interfaces with proper types.

- **[budget-api/src/notifications/notifications.controller.ts:37-38]** *(Conventions / Swagger)* -- The `@ApiResponse` for `GET /notifications` specifies `type: NotificationDto` but the actual return type is `PaginatedDataDto<NotificationDto>`. Swagger will not correctly document the paginated wrapper. **Fix:** Use a custom Swagger schema or `@ApiExtraModels()` with `@ApiResponse({ schema: ... })` to properly document the paginated response.

- **[budget-api/src/notifications/entities/notification.entity.ts:10-38]** *(Conventions)* -- The `Notification` entity class does not extend `BaseSchema`, unlike the approach described in CLAUDE.md (all entities use `BaseSchema` for `_id -> id` transform). It only adds `AuditableSchema` via `.add()`. The `_id -> id` transform works because `toObject()` + `plainToClass` handles it, but it deviates from the documented pattern. **Fix:** Consider extending `BaseSchema` or adding the transform option to the schema configuration.

- **[budget-ui/src/lib/components/notification-item.svelte:54-55]** *(Accessibility)* -- The `svelte-ignore` comments suppress `a11y_click_events_have_key_events` and `a11y_no_static_element_interactions` warnings. The notification item div is clickable but not keyboard-accessible (no `role="button"`, no `tabindex`, no `onkeydown` handler). **Fix:** Add `role="button"`, `tabindex="0"`, and an `onkeydown` handler that triggers `handleClick` on Enter/Space.

- **[budget-api/src/notifications/jobs/bill-scanner.job.ts:62-63]** *(Code Quality)* -- `bill.getInstances()` is called with a 30-day lookback (`now - 30 days`), but overdue bills could be older than 30 days. The lookback period should be configurable or documented. **Fix:** Make the lookback period configurable or add a comment explaining the 30-day window choice.

- **[budget-ui/src/lib/components/notification-preferences-dialog.svelte:51-63]** *(Reactivity)* -- The `$effect` that resets form state from `preferences` uses deep clone via `JSON.parse(JSON.stringify(...))` for channels. While functional, this runs on every `preferences` change including after save (since `handleSavePreferences` updates `preferences`), causing a redundant re-render. **Fix:** Use `$effect` with a guard to only reset when the dialog opens, or use `structuredClone` instead of JSON round-trip.

- **[budget-api/src/notifications/services/notification-dispatcher.service.ts:32-33]** *(Code Quality)* -- The default for `emailEnabled` is `false` when no channel config exists (`channelConfig?.email ?? false`). This means new notification types added to the enum but not to the default channel schema will have email disabled by default. This is likely intentional, but worth documenting. **Fix:** Add a code comment explaining the default behavior for new/unknown notification types.

- **[budget-ui/src/routes/api/notifications/+server.ts:13]** *(Error Handling)* -- `await response.json()` in the error path could throw if the API returns a non-JSON error response (e.g., 502 from a proxy). Same pattern in all 6 proxy routes. **Fix:** Wrap the error response parsing in a try-catch: `const data = await response.json().catch(() => ({ message: 'Unknown error', statusCode: response.status }))`.

- **[budget-api/src/notifications/entities/notification-preference.entity.ts:46]** *(TypeScript)* -- `channels` is typed as `Record<string, { inApp: boolean; email: boolean }>` but the schema defines specific keys matching `NotificationType`. The type should use `Partial<Record<NotificationType, { inApp: boolean; email: boolean }>>` for better type safety. **Fix:** Update the type to use `NotificationType` as keys.
