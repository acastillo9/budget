# Code Review: Receipt & Document Attachments for Transactions

## Critical

- **[budget-api/src/transactions/transactions.service.ts:593]** *(Data Consistency)* -- `removeAllByTransaction()` is called inside the DB transaction session callback, but the method itself does not accept or use a `session` parameter. If the surrounding transaction rolls back (e.g., `findOneAndDelete` succeeds but a later step fails), the attachment documents will already be deleted from MongoDB without being part of the atomic transaction. The same issue exists on lines 639-643 in `removeTransfer()`. **Fix:** Add an optional `session?: ClientSession` parameter to `removeAllByTransaction()` and pass it to the `deleteMany()` call. Move the S3 delete calls to after the DB transaction commits (or use a post-commit hook pattern) to avoid orphaned S3 objects on rollback.

- **[budget-api/src/attachments/attachments.service.ts:186-190]** *(Data Consistency)* -- The `countByTransactions` aggregation compares `transaction: { $in: transactionIds.map((id) => id) }` but `transactionIds` are strings, while the `transaction` field stores ObjectIds. In MongoDB aggregation pipelines (unlike Mongoose `find`), automatic string-to-ObjectId casting does not occur. This will return zero counts for all transactions. **Fix:** Map the IDs through `new mongoose.Types.ObjectId(id)` before passing to `$in`. The same issue applies to the `workspace` field on line 189.

- **[budget-api/src/attachments/attachments.service.ts:30-99]** *(Security - SSRF/Path Traversal)* -- The `file.originalname` is used directly as the `filename` field stored in the database (line 77). While the S3 key is safely generated with a UUID, a malicious `originalname` containing path traversal sequences (e.g., `../../etc/passwd`) could be stored and later displayed to users in the UI. **Fix:** Sanitize `file.originalname` by stripping path components (e.g., `path.basename(file.originalname)`) before storing.

## Warning

- **[budget-api/src/attachments/attachments.service.ts:36-45]** *(DRY)* -- The MIME type validation is duplicated: once in the controller's `FileInterceptor` fileFilter (controller line 71-83) and again in the service (lines 40-45). If the allowed types list changes, both locations must be updated. **Fix:** Extract `ALLOWED_MIME_TYPES` to a shared constant file (e.g., `attachments.constants.ts`) imported by both controller and service, or remove the redundant service-level check since the controller interceptor already rejects disallowed types.

- **[budget-api/src/attachments/attachments.service.ts:30-99]** *(i18n)* -- Error messages in the attachments service are hardcoded in English (e.g., `'File is required'`, `'Unsupported file type. Allowed: JPG, PNG, WebP, PDF.'`, `'Maximum of 5 attachments per transaction.'`). The API has i18n translation files (`en/attachments.json`, `es/attachments.json`) but `I18nService` is never injected or used in `AttachmentsService`. Other services in the codebase (e.g., `AuthService`, `WorkspacesService`) use `this.i18n.t()` for user-facing messages. **Fix:** Inject `I18nService` into `AttachmentsService` and replace hardcoded strings with `this.i18n.t('attachments.unsupportedType')`, etc.

- **[budget-api/src/attachments/s3.service.ts:18]** *(Error Handling)* -- `configService.get<string>('AWS_S3_BUCKET')` returns `undefined` when the env var is missing, but the code does not validate this. The S3Client will be constructed with `undefined` credentials and bucket, leading to cryptic AWS SDK errors at runtime rather than a clear startup failure. **Fix:** Use `configService.getOrThrow<string>('AWS_S3_BUCKET')` (available in NestJS v9+) or add explicit null checks with descriptive error messages in the constructor.

- **[budget-api/src/attachments/entities/attachment.entity.ts:42-43]** *(Schema Design)* -- The `workspace` field is marked as optional (`workspace?: WorkspaceDocument`) without `required: true`, unlike the `transaction` and `user` fields. Since workspace isolation is critical for multi-tenant security (all queries filter by workspace), this field should be required to prevent documents from being created without workspace scoping. **Fix:** Add `required: true` to the `@Prop` decorator for `workspace`.

- **[budget-ui/src/lib/components/attachment-list.svelte:46]** *(UX - Wrong Error Message)* -- When a download fails, the error toast displays `$t('transactions.attachments.uploadError')` (upload error) instead of a download-specific error message. **Fix:** Add a `downloadError` i18n key (e.g., `"Error downloading attachment. Please try again."`) and use it here, or reuse `deleteError` if a generic error is acceptable.

- **[budget-ui/src/lib/components/create-transaction-form.svelte:49-55]** *(Reactivity)* -- The `$effect` that calls `loadAttachments(formData.id)` will re-run every time any property of `formData` changes (since `formData.id` creates a dependency on the entire `formData` object in Svelte 5). This could cause unnecessary API refetches. The same issue exists in `create-transfer-form.svelte:49-55`. **Fix:** Extract `formData.id` into a local derived variable and use that in the effect, or use `$effect` with an explicit dependency.

- **[budget-ui/src/lib/locales/es.json:486,492,495,496]** *(i18n Quality)* -- Several Spanish translations are missing diacritical marks: `"aqui"` should be `"aqui"` (though actually "aqui" needs accent: `"aquí"`), `"Estas seguro"` should be `"¿Estas seguro"` -> `"¿Estás seguro"`, `"accion"` -> `"acción"`, `"Maximo"` -> `"Máximo"`, `"transaccion"` -> `"transacción"`, `"tamano"` -> `"tamaño"`. **Fix:** Add the correct accents and special characters to the Spanish translations.

## Suggestion

- **[budget-api/src/attachments/attachments.service.ts:181-205]** *(Dead Code)* -- The `countByTransactions` method is defined but never called anywhere in the codebase. The `attachmentCount` field was added to `TransactionDto` but is never populated in `TransactionsService.findAll()`. **Fix:** Either wire up `countByTransactions` in `TransactionsService.findAll()` to populate `attachmentCount` on each transaction DTO, or remove both the method and the DTO field if this feature is deferred.

- **[budget-api/src/attachments/attachments.service.ts:166-178]** *(Performance)* -- `removeAllByTransaction` fetches all attachment documents, loops through them to delete S3 objects one-by-one sequentially, then does a `deleteMany`. For transactions with many attachments, this is slow. **Fix:** Use `Promise.all()` for the S3 delete calls to parallelize them: `await Promise.all(attachments.map(a => this.s3Service.delete(a.s3Key)))`.

- **[budget-api/src/attachments/attachments.controller.ts:29-34]** *(DRY)* -- `ALLOWED_MIME_TYPES` is defined separately in both the controller and the service. `MAX_FILE_SIZE` is defined in the controller but `MAX_ATTACHMENTS_PER_TRANSACTION` is in the service. These related constants should be colocated. **Fix:** Create an `attachments.constants.ts` file exporting all attachment-related constants.

- **[budget-api/src/attachments/attachments.controller.ts:86-97]** *(Validation)* -- The `transactionId` and `attachmentId` path parameters are not validated as MongoDB ObjectIds. Invalid IDs (e.g., `"abc"`) will cause a Mongoose `CastError` resulting in a 500 Internal Server Error instead of a clean 400. The existing `TransactionsController` has the same pattern (no validation), so this is consistent but still worth noting. **Fix:** Use a `ParseMongoIdPipe` or add `@IsMongoId()` validation.

- **[budget-ui/src/lib/components/create-transaction-form.svelte & create-transfer-form.svelte]** *(DRY)* -- The attachment-related code (state variables, `loadAttachments`, `handleFileSelect`, `handleDeleteAttachment`, and the template block) is duplicated verbatim between the two form components (~60 lines each). **Fix:** Extract the shared attachment logic into a reusable composition function (e.g., `createAttachmentState()`) or a dedicated `TransactionAttachments.svelte` wrapper component.

- **[budget-api/src/attachments/dto/attachment.dto.ts]** *(Conventions)* -- The `AttachmentDto` does not use `@Exclude()`/`@Expose()` decorators or `class-transformer` features. The `plainToClass` call in the service relies on the default behavior which copies all properties. The `_id` to `id` transform comes from `AuditableSchema`, but the DTO does not exclude internal fields like `s3Key`, `transaction`, `user`, `workspace`. These leak into the API response. **Fix:** Add `@Exclude()` at the class level and `@Expose()` on each public field, or verify that `toObject()` with the `BaseSchema` transform already strips internal fields.

- **[budget-api/test/attachments.e2e-spec.ts:63]** *(Test Robustness)* -- The `createTestAppWithMockedS3` function duplicates the global pipe and interceptor setup from `main.ts` / `test-app.factory.ts`. If the factory is updated (e.g., adding new global middleware), this test setup will diverge. **Fix:** Use the existing `test-app.factory.ts` pattern and add the S3 mock override, or extract the common setup into a shared helper.

- **[budget-ui/e2e/attachments.spec.ts]** *(Test Cleanup)* -- Temporary directories (`.tmp-test-val`, `.tmp-test-large`, etc.) are created inside `process.cwd()`. While `finally` blocks clean them up, test failures or interruptions could leave them behind. **Fix:** Use `os.tmpdir()` or Playwright's built-in temporary file handling instead of creating directories in the project root.
